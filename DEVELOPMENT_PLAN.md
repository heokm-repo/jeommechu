# 점메추 MVP 개발 계획서

작성일: 2026-06-24  
현재화: 2026-06-28

## 1. 문서 목적

이 문서는 현재 저장소 기준으로 점메추 MVP의 구현 상태와 남은 작업 우선순위를 정리한다. 이전 계획서에 남아 있던 오래된 갭 목록은 현재 코드와 맞지 않는 항목이 많아, 이 문서에서는 다음 세 가지를 기준으로 다시 정리한다.

- 이미 구현된 핵심 흐름은 완료 항목으로 분리한다.
- 실사용 MVP에 반드시 필요한 안정화 작업을 우선순위로 둔다.
- 개인화, 공유, 상세 UX 같은 확장 기능은 후순위로 둔다.

## 2. 제품 목표

점메추는 사용자의 감정과 상황을 입력받아 Kakao Local API 검색에 적합한 음식 검색어를 만들고, 현재 위치 또는 선택 지역 주변의 음식점을 추천하는 서비스다.

MVP의 성공 기준은 다음 흐름이 끊기지 않는 것이다.

1. 사용자가 자유 텍스트 또는 감정/상황 태그로 현재 상태를 입력한다.
2. Gemini가 Kakao 검색용 음식 카테고리 또는 검색어를 만든다.
3. 백엔드는 해당 검색어와 위치/반경으로 Kakao Local API를 호출한다.
4. 음식점 정보와 지도 좌표를 결과 카드와 주변 맛집 목록으로 보여준다.
5. 검색 기록, 추천 결과, 찜, 방문, 피드백 데이터를 사용자에게 연결한다.
6. 게스트 사용자가 회원가입해도 기존 기록을 이어서 사용할 수 있다.

## 3. 현재 구현 상태

### 프론트엔드

- `app.js`는 정적 HTML 조각을 로딩하는 SPA 라우터로 동작한다.
- `scripts/state.js`는 `user`, `userId`, 위치, 검색 상태, 추천 결과, 찜, 히스토리를 `localStorage`에 저장하고 복원한다.
- `pages/home.html`과 관련 스크립트는 직접 입력, 태그 선택, 지역 선택, 현재 위치 가져오기, 검색 반경 선택을 제공한다.
- `scripts/api.js`는 백엔드 API 래퍼를 제공한다.
- `scripts/auth.js`는 게스트 생성, 로그인, 회원가입, 로그아웃을 실제 API로 호출한다.
- 추천 결과 화면은 추천 검색어, 추천 사유, 날씨 정보, 1순위 식당, 정적 지도, 피드백 버튼을 표시한다.
- 주변 맛집 화면은 Kakao 지도 마커와 음식점 리스트를 표시하고, 찜/방문 기록 액션을 연결한다.
- 찜, 히스토리, 프로필, 로그인, 회원가입, 에러 화면이 실제 API 기반으로 연결되어 있다.

### 백엔드

- `backend/server.py`는 표준 라이브러리 `ThreadingHTTPServer` 기반 API/정적 파일 서버다.
- `.env`를 로드하고, 시작 시 MySQL DB와 스키마를 초기화한다.
- DB 접근은 PyMySQL과 명시적 SQL을 사용한다.
- MySQL 스키마는 `backend/schema.mysql.sql`과 `backend/schema.sql`에 유지되어 있다.
- Kakao Local API 주소 검색, 키워드 검색, 음식점 카테고리 검색을 수행한다.
- Gemini를 사용해 Kakao 검색어를 계획한다.
- 기상청 초단기 실황 API를 통해 현재 날씨 문맥을 추천 입력에 포함한다.
- 외부 API 키 누락, Kakao/Gemini/날씨 호출 실패, Gemini 응답 검증 실패 시 rule-based/mock/fallback 추천을 만들지 않고 오류를 반환한다.
- 게스트 생성, 이메일 회원가입, 로그인, 로그아웃, 세션 쿠키 발급/폐기가 구현되어 있다.
- `auth_sessions`와 `jeommechu_session` 쿠키를 통해 세션 사용자를 확인한다.
- 찜, 히스토리, 방문, 피드백, 추천 상세 조회는 세션 소유자 검사를 거친다.
- 추천 검색 시 검색 로그, AI 응답, 추출 키워드, 음식점 스냅샷, 추천 결과, 이벤트 로그를 DB에 저장한다.

### DB

현재 스키마의 핵심 테이블은 다음과 같다.

- `users`: 게스트/회원 사용자와 이메일 인증 정보
- `auth_sessions`: 세션 토큰 해시와 만료/폐기 정보
- `search_logs`: 사용자별 추천 검색 기록
- `ai_responses`: Gemini 요청/응답 스냅샷
- `ai_extracted_keywords`: 감정, 음식, 상황, 장소 키워드
- `restaurants`: Kakao 음식점 스냅샷
- `restaurant_search_cache`: Kakao 검색 실행 스냅샷
- `search_recommendations`: 검색별 추천 결과
- `recommendation_logs`: `SEARCH`, `CLICK`, `FAVORITE`, `VISIT`, `FEEDBACK` 이벤트 로그
- `user_favorites`: 사용자별 찜
- `visit_histories`: 방문 기록
- `regions`: 선택/저장된 지역 정보

DB 원칙은 유지한다.

- MySQL 8.x, InnoDB, `utf8mb4`를 기준으로 한다.
- UUID는 MVP 기준 `CHAR(36)`으로 저장한다.
- JSON 필드는 MySQL `JSON` 타입을 사용한다.
- 시간 컬럼은 UTC 기준 `DATETIME(6)`으로 통일한다.
- 음식점 탐색의 원천은 DB가 아니라 Kakao Local API다.
- DB의 음식점 데이터는 추천 이력, 사용자 행동, 분석을 위한 스냅샷이다.

## 4. 현재 완료된 핵심 기능

### 완료

- MySQL 기준 스키마와 서버 초기화
- 게스트 사용자 생성과 세션 쿠키 발급
- `localStorage` 기반 프론트 상태 복원
- 이메일 회원가입/로그인/로그아웃
- 게스트 사용자 회원 전환
- 세션 소유자 검사 기반 사용자 데이터 보호
- Gemini 기반 Kakao 검색어 생성
- Kakao Local API 기반 음식점 검색
- 날씨 문맥을 추천 입력에 포함
- 추천 결과와 음식점 스냅샷 DB 저장
- 찜 저장/조회/삭제
- 추천 히스토리 조회와 상세 복원
- 방문 기록 저장/조회
- 추천 피드백 저장
- API/SDK/설정 오류를 `error.html`에서 표시
- 기본 백엔드 단위 테스트

### 부분 완료 또는 보강 필요

- 세션 인가는 구현되어 있지만, 권한 실패/세션 만료/타 사용자 접근에 대한 테스트가 부족하다.
- 백엔드 테스트는 존재하지만 외부 API stub 기반의 end-to-end 테스트는 부족하다.
- 프론트 smoke test는 아직 없다.
- 추천 품질은 Gemini 검색어와 Kakao 결과 순서에 많이 의존한다.
- 추천 점수는 현재 개인화 점수가 아니라 순위 기반 점수에 가깝다.
- 배포 환경의 CORS, 비밀키, DB 백업, 로그 마스킹 정책은 아직 문서화가 부족하다.
- 개발용 DB 재생성/seed 절차가 명확하지 않다.

## 5. 남은 필수 작업

실사용 가능한 MVP로 보기 위해 우선 처리할 항목이다.

### P0. 인증/인가 검증 강화

현재 세션 기반 소유자 검사는 구현되어 있다. 다음 단계는 이 동작이 깨지지 않도록 테스트를 고정하는 것이다.

- 세션 없는 요청이 찜/히스토리/방문/피드백에 접근하면 401을 반환하는지 검증한다.
- 다른 사용자의 `user_id` 또는 `search_log_id`로 접근하면 403을 반환하는지 검증한다.
- 게스트 생성 후 발급된 세션으로 추천, 찜, 히스토리 조회가 이어지는지 검증한다.
- 회원가입 시 기존 게스트 세션의 기록이 같은 사용자에게 유지되는지 검증한다.
- 세션 만료/로그아웃 후 보호 API 접근이 차단되는지 검증한다.

### P0. 프론트 smoke test 추가

브라우저에서 깨지기 쉬운 핵심 흐름을 자동 또는 반자동으로 검증한다.

- 홈 화면 렌더링
- 게스트 생성과 `userId` 저장
- 추천 버튼 클릭 시 payload 구성
- 추천 성공 후 결과 화면 이동
- 주변 맛집 목록 렌더링
- 찜 토글 후 새로고침 유지
- 히스토리 화면에서 방금 추천한 기록 확인
- API 오류 시 `error.html` 표시

### P0. DB 재현성 확보

새 개발 환경에서도 같은 상태를 만들 수 있어야 한다.

- 개발용 DB 삭제/재생성 절차 문서화
- UTF-8 기준 seed 데이터 작성
- seed 없이도 최소 기능이 동작하는지 확인
- 외부 API 키가 없는 환경에서 추천이 의도대로 실패하는지 확인
- DB 마이그레이션 또는 스키마 적용 책임을 서버 시작 로직과 분리할지 결정

### P0. 운영 기본 정책 문서화

배포 전 최소 운영 기준을 정한다.

- CORS 허용 origin 제한
- API 키 저장/회전 방식
- DB 백업과 복구 절차
- 감정/상황 텍스트 로그의 개인정보 마스킹 정책
- 오류 로그에 API 키, 세션 토큰, 비밀번호가 남지 않는지 점검
- 운영 환경에서 `.env` 누락 시 어떤 오류를 보여줄지 확정

## 6. 후순위 기능

아래는 제품 완성도를 높이지만, 현재 MVP 필수 조건은 아니다.

### 추천 품질과 개인화

- 감정 분류 체계 정리: 스트레스, 우울, 피로, 더위, 기쁨, 평범함 등
- 감정 -> 메뉴 키워드 후보 -> 제외 조건 -> 설명 템플릿 매핑
- Gemini prompt 입력/출력 스키마 버전 관리
- Gemini 응답 JSON 검증 강화
- 찜/방문/피드백 기반 사용자별 취향 프로파일
- 과거 행동 기반 추천 순위 가중치
- 계절 메뉴 키워드
- 날씨별 메뉴 가중치

### 화면 완성도

- 공유 가능한 감정+음식 결과 카드
- Web Share API와 링크 복사 fallback
- 맛집 상세 바텀시트
- 예산 필터
- 동행 상황 필터 고도화
- 사진, 평점, 대기 시간 같은 외부 데이터 출처 확정 후 선택 표시

### 성능과 구조 개선

- MySQL connection pool 도입 여부 결정
- DB 접근 계층 정리
- 외부 API 클라이언트 재시도/timeout 정책 정리
- API 계약 문서 자동화 또는 예제 보강
- 프론트 모듈 구조 정리

## 8. 검증 시나리오

MVP 회귀 검증은 다음 시나리오를 기준으로 한다.

1. 서버 시작 후 `GET /api/health`가 `{"ok": true}`를 반환한다.
2. 새 브라우저에서 홈 진입 시 게스트 `user_id`와 세션 쿠키가 생성된다.
3. "스트레스 받아서 매운 국물이 먹고 싶다"를 입력해 추천 요청을 보낸다.
4. MySQL DB에 검색 로그, AI 응답, 추출 키워드, Kakao 음식점 스냅샷, 추천 결과, 이벤트 로그가 저장된다.
5. 결과 화면에서 추천 검색어, 추천 사유, 날씨 문맥, 1순위 음식점이 표시된다.
6. 주변 맛집 화면에서 Kakao 지도와 음식점 목록이 표시된다.
7. 찜을 누른 뒤 새로고침해도 찜 상태가 유지된다.
8. 방문 기록을 남기면 DB에 저장된다.
9. 추천 피드백을 남기면 `recommendation_logs`에 `FEEDBACK` 이벤트가 저장된다.
10. 추천 기록 화면에서 방금 검색한 기록이 최신순으로 보인다.
11. 게스트 상태에서 회원가입한 뒤 기존 추천 기록과 찜이 유지된다.
12. 세션 없이 다른 사용자의 찜/히스토리/추천 상세에 접근하면 차단된다.
13. Kakao/Gemini/날씨 API 키가 없거나 호출이 실패하면 대체 추천을 만들지 않고 오류 화면으로 이동한다.

## 9. 결정 필요 사항

- 운영 배포 전에 session cookie만으로 충분한지, CSRF 방어를 추가할지 결정한다.
- 외부 API 키 누락 시 홈 진입 단계에서 막을지, 추천 요청 시점에 막을지 정책을 정한다.
- 날씨 API 실패가 전체 추천 실패가 되어야 하는지, 날씨만 제외하고 진행할지 제품 정책을 정한다.
- Kakao Local API가 제공하지 않는 사진, 평점, 대기 시간은 계속 숨길지, 별도 데이터 소스를 붙일지 결정한다.
- 개인정보가 포함될 수 있는 감정/상황 텍스트를 얼마나 오래 저장할지 결정한다.
- MySQL 마이그레이션 도구를 도입할지, 현재처럼 스키마 파일 적용 방식으로 유지할지 결정한다.
